## 1. 제품 개요

### 1.1 문제 정의

LLM을 도입하려는 기업과 개발자가 폭발적으로 늘고 있다. 그러나 실제 도입 과정에서 공통적으로 겪는 문제가 있다.

- **선택의 과부하:** 상용/오픈소스 합쳐 수백 개의 모델이 존재하고, 매주 신규 모델이 출시된다. 어떤 모델이 우리 업무에 맞는지 판단할 기준이 없다.
- **비용 예측 불가:** 모델마다 가격 체계가 다르고, 프롬프트 캐싱/Batch API/양자화 등 비용 절감 옵션이 다양하지만 이를 종합적으로 시뮬레이션할 수 없다.
- **산업 특화 기준 부재:** 범용 벤치마크(MMLU, GPQA 등)는 존재하지만, "고객 상담 챗봇에 최적인 모델"이나 "법률 문서 분석에 강한 모델"처럼 업무 맥락에 맞는 비교 기준이 없다.
- **인프라 정보 분산:** 오픈소스 모델을 셀프호스팅하려면 GPU 스펙, VRAM, 서빙 프레임워크 정보가 필요하지만 모델마다 흩어져 있다.
- **실전 체감 부재:** 벤치마크 점수만으로는 실제 응답 품질을 알 수 없다. 동일 프롬프트로 여러 모델을 나란히 비교해볼 도구가 없다.

### 1.2 제품 비전

> **어떤 산업, 어떤 업무든 — "우리 상황에 맞는 LLM 조합"을 10분 안에 찾고, 비용까지 시뮬레이션할 수 있는 플랫폼.**

단순한 모델 스펙 비교를 넘어, 사용자의 산업/업무/예산/인프라 조건을 입력하면 최적의 모델 조합과 운영 전략을 제안하는 의사결정 도구를 만든다.

### 1.3 목표 사용자

| 사용자 | 핵심 니즈 | 주요 사용 기능 |
|--------|-----------|---------------|
| 사업 기획자 / PM | 업무별 최적 모델 선정, 도입 비용 근거 확보 | BVA 리포트, 산업별 추천, 비용 시뮬레이터 |
| 개발자 / MLOps | 모델 스펙 비교, 셀프호스팅 인프라 설계, 실전 테스트 | Playground, 인프라 가이드, VRAM 계산기, 벤치마크 비교 |
| 경영진 / 의사결정자 | 요약된 비교 근거, 비용 전망, ROI 판단 | BVA 리포트, 비용 차트 |
| 프리랜서 / 1인 개발자 | 예산 내 최적 모델 빠른 탐색 | 가격순 정렬, 가성비 랭킹, 로컬 실행 가이드 |

### 1.4 경쟁 환경과 차별점

| 기존 도구 | 한계 | 우리의 차별점 |
|-----------|------|--------------|
| Artificial Analysis | 벤치마크/속도 비교 중심. 업무 맥락 없음 | BVA 8차원 평가 + 산업별 추천 |
| OpenRouter Rankings | 토큰 사용량 랭킹만 제공 | Playground 실전 비교 + 비용 시뮬레이터 |
| PricePerToken | 가격 비교 전용 | 인프라 스펙 + 한국어 등 다국어 평가 통합 |
| 각 벤더 공식 문서 | 자사 모델만 다룸 | 50개+ 모델 크로스벤더 통합 비교 |

### 1.5 핵심 결정사항

| 결정 | 내용 | 근거 |
|------|------|------|
| 인증 제외 | 전체 공개 도구 | 도구 용도, 사용 장벽 최소화 |
| 관리자 CRUD UI 제외 | 시드 데이터(JSON) + import 스크립트로 대체 | 개발 비용 절감, 데이터 품질 관리 용이 |
| Llama/Meta 모델 제외 | 수집 대상에서 제외 | 벤치마크 대비 실사용 선호도 괴리 |

## 2. 기능 요구사항

### F1. 모델 데이터베이스

**F1.1 모델 정보 구조**

각 모델은 6개 레이어의 정보를 포함한다.

**L1. 식별 정보:**
- 기본: 모델명, slug, 유형(상용/오픈소스), 출시일, 라이선스
- 분류: 성능 등급(flagship/mid/light), 용도 태그(reasoning, coding, multimodal 등)
- 계보: 모델 패밀리, 변형(variant)
- 상태: active, preview, deprecated, scheduled-deprecation
- 프로바이더: providers 컬렉션 FK 참조

**L2. 기술 스펙:**
- 아키텍처: 전체 파라미터, 활성 파라미터(MoE), 구조(Dense/MoE)
- 컨텍스트: 최대 입력 토큰, 최대 출력 토큰
- 양식: 입력 모달리티(text, image, audio 등), 출력 모달리티
- 기능: function calling, structured output, streaming, vision, tool use, fine-tuning, batch API, thinking mode, system prompt
- 기타: 학습 데이터 기준일, 지원 언어 목록

**L3. 벤치마크 (11개):**

| 벤치마크 | 카테고리 |
|----------|----------|
| MMLU | 지식 |
| GPQA, AIME, HLE | 추론 |
| SWE-bench | 코딩 |
| KMMLU, MGSM | 한국어 |
| TruthfulQA | 신뢰성 |
| BFCL | 도구호출 |
| IFEval | 명령어수행 |
| RULER | 긴문서처리 |

- Model에 최신 점수 캐시 유지 (조회 성능)
- model_benchmarks 컬렉션에 시점별 이력 관리 (출처, 측정일, methodology 포함)

**L4. 비용:**
- Model에 현재 가격 캐시 (input/output 단가, pricing_type)
- model_pricing 컬렉션에 가격 이력 (effective_from/to, 배치/캐싱 할인, 셀프호스팅 비용)

**L5. 운영 특성:**
- 컴플라이언스: SOC2, HIPAA, GDPR, 온프레미스, 데이터 제외
- 성능 지표: 평균 TPS, TTFT (null 허용, 실측 데이터 축적)
- 지역 제한: regions 배열 (null 허용)

**인프라 (OSS 전용):**
- 최소 GPU 구성, VRAM (FP16/FP8/INT8/INT4/GGUF 양자화별)
- 권장 서빙 프레임워크, 예상 TPS

**F1.2 프로바이더 정보**

모델 제공사를 별도 컬렉션으로 관리한다.
- 기본: 이름, 영문명, 유형(commercial/commercial+oss/oss)
- 상세: 본사 소재지, 설립연도, 웹사이트, API 엔드포인트, 설명
- UI: 색상 코드 (차트/배지 표시용)

**F1.3 데이터 관리**

- JSON 시드 데이터 + import 스크립트로 데이터 관리
- 각 데이터에 "최종 확인일" 표시
- 가격 변경 시 model_pricing 컬렉션에 이력 보존
- `--force` 모드: 전체 drop 후 재삽입 (스키마 변경 시)
- 향후 외부 API 자동 수집 연동을 고려한 구조

### F2. 모델 탐색

**F2.1 테이블 뷰**

- 전체 모델 한 화면 조회
- 필터: 유형, 성능 등급, 제공사, 라이선스, 가격 범위, 태그, 기능(capabilities)
- 모든 수치 컬럼 정렬
- 모델명/제공사 텍스트 검색
- 컬럼 표시 커스터마이즈

**F2.2 파라미터 사이즈별 카드 뷰**

- 오픈소스 모델을 등급별 그룹핑 (flagship/mid/light)
- 각 카드에 인프라 요구사항 강조 (파라미터, 아키텍처, GPU, VRAM, 라이선스 배지)
- 그룹 헤더에 해당 등급 공통 하드웨어 가이드

**F2.3 모델 상세 페이지**

- 전체 스펙, 벤치마크 시각화, 인프라 가이드, 비용 예시
- 유사 모델 자동 추천 (같은 등급/가격대)
- 개별 URL로 검색엔진 인덱싱

### F3. 모델 비교

**F3.1 비교 선택**

- 레이더 차트 (선택한 평가 항목 기준)

### F4. 비용 시뮬레이터 (Phase 2)

**F4.1 API 비용 계산**

입력:
- 일 평균 요청 건수
- 평균 Input/Output 토큰 수
- 월 운영 일수
- 프롬프트 캐싱 적용률 (슬라이더)
- Batch API 활용 비율 (슬라이더)

출력:
- 선택 모델들의 월간/연간 비용 바 차트
- 파라미터 변경 시 실시간 갱신

**F4.2 셀프호스팅 손익분기점**

입력:
- GPU 임대 시간당 단가 (또는 구매 가격 + 감가)
- 일 가동시간
- 월 부대비용 (전기, 관리 인건비)

출력:
- "API vs 셀프호스팅" 월간 비용 비교
- 요청량 증가에 따른 손익분기점 차트
- "일 N건 이상이면 셀프호스팅이 유리" 메시지

**F4.3 라우팅 비용 시뮬레이션**

- 단일 모델 처리 vs 다단계 라우팅 처리 비용 비교
- 라우팅 비율 조절 (예: 70% 저가, 20% 중급, 10% 고급)
- 라우팅 적용 시 절감률 자동 계산

### F5. 인프라 가이드

**F5.1 VRAM 계산기 (Phase 2)**

- 입력: 파라미터 수, 양자화 방식(FP16/FP8/INT8/INT4/GGUF), 컨텍스트 길이
- 출력: 필요 VRAM + 적합 GPU 매핑
- MoE 모델 안내 (전체 가중치 로드, 활성 파라미터만 연산)

**F5.2 GPU 레퍼런스**

- 주요 GPU별 VRAM, 가격, 특성 테이블 (11개 GPU, 3개 카테고리)
- 역방향 조회: "이 GPU로 돌릴 수 있는 모델 목록"
- GPU 호환성 자동 계산 (양자화 수준별 VRAM 매칭)

**F5.3 서빙 프레임워크 가이드 (Phase 2)**

- vLLM, Ollama, llama.cpp, SGLang 비교
- 모델별 권장 프레임워크 매핑

### F6. BVA (Business Value Assessment)

**F6.1 평가 체계**

8개 차원의 벤치마크 기반 평가:

| 차원 | 공식 | 활용 |
|------|------|------|
| reasoning | GPQA 40% + AIME 30% + HLE 30% | 복잡한 문제 해결 |
| korean | KMMLU 70% + MGSM 30% | 한국어 능력 |
| coding | SWE-bench 100% | 코딩/엔지니어링 |
| knowledge | MMLU 70% + GPQA 30% | 지식 범위 |
| reliability | TruthfulQA 100% | 환각 방지/신뢰성 |
| tool_use | BFCL 100% | 도구 호출/API 연동 |
| instruction | IFEval 100% | 명령어 수행력 |
| long_context | RULER 100% | 긴 문서 처리 |

+ cost (비용 효율성): OSS = 100, 상용 = max(0, 100 - output/60 * 100)

- null 벤치마크 처리: 가용 벤치마크로 가중치 재정규화
- 프로바이더 다양성: max 2개/프로바이더 제한

**F6.2 BVA 입력 (사용자 프로필)**

- 산업, 업무 유형
- 월간 토큰 볼륨 (under-10k / 10k-100k / 100k-1m / over-1m)
- 사용 언어 (ko, en, ja, zh)
- 톤 (formal, casual, technical)
- 보안 요구사항 (SOC2, HIPAA, GDPR, 온프레미스, 데이터 제외)

**F6.3 BVA 리포트**

출력:
- 상용/OSS 분리 랭킹 (각 상위 5개)
- 차원별 점수 breakdown (투명한 근거)
- 컴플라이언스 체크 결과
- 비용 추정 (월간 토큰 기반)

**F6.4 산업별 프리셋**

- 12개 산업/업무 프리셋 (5개 카테고리: 금융, 개발/IT, 고객 서비스, 영업/마케팅, 이커머스)
- 각 프리셋에 9개 차원 가중치 + 정적 추천 모델
- 사용자 커스터마이즈 가능

**F6.5 BVA 방법론 페이지**

- 평가 차원 설명, 벤치마크별 해석, 점수 산출 로직 공개

**F7.3 세션 관리**

- 세션 저장/복원 (MongoDB)
- 세션 목록 조회, 삭제
- 보안: Zod 검증, DB 모델 허용목록, IP 기반 Rate Limiting (20 req/min)

## 5. 데이터 구조

### 5.1 컬렉션 구조 (9개)

```
Core
├── providers              ← 프로바이더 마스터 (풀 스펙)
└── models                 ← 모델 마스터 (L1~L5 통합, 벤치마크/가격 캐시)

Model 하위
├── model_benchmarks       ← 벤치마크 점수 이력 (Long format)
└── model_pricing          ← 가격 이력 (effective_from/to)

참조 데이터
├── ref_benchmarks         ← 벤치마크 정의 (11개)
└── ref_gpus               ← GPU 스펙 참조 (11개)

BVA
├── bva_dimensions         ← 평가 차원 정의 (8개)
└── bva_presets            ← 산업별 프리셋 (12개)

Playground
└── playground_sessions    ← 채팅 세션
```

### 5.2 관계

| From | To | Cardinality | 전략 |
|------|----|-------------|------|
| providers | models | 1:N | FK (provider_id) |
| models | model_benchmarks | 1:N | FK (model_id) |
| models | model_pricing | 1:N | FK (model_id) |
| ref_benchmarks | model_benchmarks | 1:N | FK (benchmark_id) |
| ref_benchmarks | bva_dimensions.formula | 1:N | key 참조 |
| bva_dimensions | bva_presets.weights | 1:1 | key 매핑 |
| models | playground_sessions.models | N:M | 내장 참조 |
| models | ref_gpus | N:M | infrastructure.minGpu 참조 |

### 5.3 BVA 점수 산출 로직

```
BVA 점수 = Σ (차원별 벤치마크 가중점수 × 프리셋 가중치)

예시 — "코드 리뷰" 프리셋 (weights):
  reasoning: 25, korean: 10, coding: 30, knowledge: 5,
  reliability: 5, tool_use: 10, instruction: 10, long_context: 5,
  cost: 0  (합계 100)

모델 A의 coding 차원 점수 = SWE-bench 점수 (예: 72.3)
→ 기여분 = 72.3 × 0.30 = 21.69
→ 전체 차원 합산 = 최종 BVA 점수
```
