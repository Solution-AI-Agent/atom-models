# BVA 기능 개발 회고록

> 작성일: 2026-03-11
> 프로젝트: Atom Models - LLM 비교 및 최적화 플랫폼

---

## 1. 프로젝트 개요

기존 산업별 추천 시스템의 평가 체계를 전면 재설계했다. 자체 점수(scores)를 제거하고 공신력 있는 벤치마크 기반의 투명한 BVA(Business Value Assessment) 평가 체계로 전환했다.

**핵심 목표:**
- 자체 scores(quality/speed/reasoning/coding/multimodal) 제거
- 벤치마크 기반 4차원(reasoning, korean, coding, knowledge) + cost 평가 체계 구축
- 고객 프로필 기반 BVA 입력폼, 비교 리포트, 방법론 페이지 추가
- 모든 점수가 출처(벤치마크)와 투명한 계산 공식으로 설명 가능

## 2. 작업 흐름

### 2.1 브레인스토밍 (설계 논의)

코딩 전 brainstorming 스킬을 활용하여 6가지 핵심 결정을 순차적으로 합의했다:

| # | 질문 | 결정 |
|---|------|------|
| 1 | 사용자 대상 | 컨설턴트 + 고객사 담당자 (셀프서비스) |
| 2 | 기존 추천과의 관계 | 기존 프리셋 흐름 위에 BVA 입력 단계를 확장 |
| 3 | 입력 방식 | 구조화된 선택지 (드롭다운, 체크박스) |
| 4 | 평가 기준 | 자체 scores 전면 제거 -> 벤치마크 기반 |
| 5 | 결과 산출물 | 상위 2~3 모델 비교 리포트 |
| 6 | 차원 설명 UI | 리포트 내 툴팁 + /methodology 독립 페이지 |

### 2.2 팀 기반 에이전트 워크플로우

TeamCreate를 활용하여 전문 에이전트 7명이 5개 트랙을 수행했다:

| 팀원 | 역할 | 트랙 | 소요 시간 (추정) |
|------|------|------|----------------|
| t1-architect | 데이터 모델 상세 설계 | T1 | ~4분 |
| t2-researcher | 벤치마크 데이터 + 컴플라이언스 조사 | T2 | ~6분 |
| t3-engine | 스코어링 엔진 재설계 (TDD) | T3 | ~5분 |
| t4a-bva-ui | BVA 신규 UI + 방법론 페이지 | T4 | ~5분 |
| t4b-ui-migration | 기존 UI scores 제거 전환 | T4 | ~5분 |
| t-dba | Architecture 팀 스키마 비교/통합 | 부가 | ~4분 |
| t5-reviewer | 통합 테스트 + 코드 리뷰 | T5 | ~3분 |

### 2.3 병렬 실행 전략

```
Phase A: T1 (설계) + T2 (리서치)     ← 병렬
Phase B: T3 (엔진)                    ← T1 의존
Phase C: T4a (BVA UI) + T4b (마이그레이션) + DBA (스키마 비교) ← 병렬
Phase D: T5 (통합 검증)               ← T3, T4 의존
```

## 3. 주요 설계 결정

### 3.1 벤치마크 데이터 현실 반영

리서치 결과, 계획했던 4개 신규 벤치마크 중 실제 채택은 1개에 그쳤다:

| 벤치마크 | 계획 | 결정 | 이유 |
|----------|------|------|------|
| KMMLU | 채택 | 채택 | KMMLU-Redux 논문에서 주요 모델 데이터 확보 |
| KoBEST | 채택 | 제외 | 상용 모델 데이터 거의 없음 (HyperCLOVA X만) |
| TruthfulQA | 채택 | 제외 | 프론티어 모델 프로바이더가 더 이상 보고 안 함 |
| HaluEval | 채택 | 제외 | 학술용, 실용적 대안(Vectara HHEM)은 버전 간 비교 불가 |

**교훈:** 벤치마크 설계 시 데이터 가용성을 먼저 조사해야 한다. "이론적으로 좋은 벤치마크"와 "실제 데이터가 있는 벤치마크"는 다르다.

### 3.2 reliability 차원 제외

TruthfulQA + HaluEval 기반으로 설계했던 "reliability(신뢰성)" 차원을 1차에서 제외했다. 데이터 없는 차원에 점수를 넣는 것은 BVA의 핵심 원칙("설명 가능한 평가")에 위배되기 때문이다. 신뢰할 수 있는 환각 벤치마크가 확보되면 추후 추가한다.

### 3.3 자체 scores vs 벤치마크

기존 자체 scores(quality: 88 등)는 출처가 불분명하여 고객에게 "왜 이 점수인가?"를 설명할 수 없었다. 벤치마크 기반으로 전환함으로써:
- "GPQA 65점" → "대학원 수준 과학 문제 정답률 65%"로 명확한 설명 가능
- 계산 공식이 투명 → "종합 추론력 = GPQA(40%) + AIME(30%) + HLE(30%)"
- /methodology 페이지에서 전체 평가 방법론 공개

## 5. 잘된 점

1. **브레인스토밍 우선**: 코딩 전 6가지 핵심 결정을 질문-답변으로 합의하여 방향 전환이 최소화됐다.
2. **리서치로 현실 반영**: 벤치마크 데이터 가용성을 사전 조사하여 없는 데이터에 의존하는 설계를 방지했다.
3. **병렬 팀 실행**: T1+T2, T4a+T4b+DBA를 병렬로 실행하여 전체 소요 시간을 단축했다.
4. **TDD 준수**: T3에서 테스트 먼저 작성 후 구현하여 스코어링 엔진의 정확성을 보장했다.
5. **설명 가능성 원칙 관철**: "데이터 없으면 넣지 않는다"는 원칙을 지켜 BVA 신뢰성을 확보했다.

---

## 6. 개선할 점

1. **백그라운드 에이전트 → TeamCreate 전환 지연**: 처음에 백그라운드 Agent로 스폰했다가 TeamCreate로 재구성하여 중복 작업이 발생했다. 팀 작업은 처음부터 TeamCreate를 사용해야 한다.
2. **벤치마크 리서치 순서**: 설계(T1) 전에 데이터 가용성 리서치(T2)를 먼저 했으면 reliability 차원 설계 후 제외하는 왕복이 없었을 것이다.
3. **Architecture 팀 스키마 사전 공유**: 개발 중반에 외부 스키마(docs/data-schema.md)가 추가되어 DBA 분석이 별도로 필요했다. 설계 단계에서 통합했으면 더 효율적이었다.
4. **시드 데이터 KMMLU 커버리지**: 87개 모델 중 KMMLU 데이터가 있는 모델이 제한적이다. 추후 데이터 보강이 필요하다.
